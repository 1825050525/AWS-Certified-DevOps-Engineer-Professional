## 問題25：Lambda カナリアデプロイ ＋ SAM ＋ CI/CD ⭐⭐ 難易度：高
**結果：不正解**

### 一言でいうと
「Lambda カナリアデプロイは SAM に任せる。CI/CD は CodePipeline + CodeBuild で組む」

### カテゴリ別ポイント

#### SAM（デプロイ戦略）
- `AWS::Serverless::Function` で Lambda を定義
- `AutoPublishAlias` ＋ `DeploymentPreference` の2プロパティを設定するだけでカナリア完成
- SAM が内部で Lambda Alias / CodeDeploy を自動管理する → 手動設定不要

#### CI/CD パイプライン
| 役割 | サービス |
|---|---|
| ソース管理 | CodeCommit |
| パイプライン制御 | CodePipeline |
| ビルド・デプロイ実行 | CodeBuild ＋ buildspec.yml |

#### 監視・ロールバック
- 各 Lambda 関数に個別の CloudWatch アラームを作成（`AWS/Lambda` の `Errors` メトリクス）
- SAM の DeploymentPreference がアラームを検知して自動ロールバック実行

### 罠
| 選択肢 | なぜ間違い |
|---|---|
| CodeDeploy 手動作成 ＋ appspec.yml | SAM が自動生成・管理するため不要（二重管理） |
| CloudWatch 複合アラーム | 粒度が粗く関数単位の精密監視ができない |

### 全体構造
```
CodeCommit → CodePipeline → CodeBuild（sam deploy）
                                  ↓
                             SAM（Lambda更新）
                                  ↓
                         カナリア（10%テスト）
                                  ↓
                    CloudWatch アラーム（エラー監視）
                                  ↓
                            自動ロールバック
```

### 覚え方
> SAM = 「カナリア＋ロールバック」を1行で書ける魔法  
> CodeDeploy・appspec.yml の手動設定は SAM では不要

## 問題31：ブルー/グリーンデプロイ（ALB ターゲットグループ切り替え） ⭐ 難易度：低
**結果：正解**

### 一言でいうと
「ブルー/グリーンの切り替えは ALB ターゲットグループで。手順は必ず"デプロイ→切り替え"の順」

### カテゴリ別ポイント

#### 正しい手順（順序が命）
```
✅ 正解：① グリーン環境でローリング再起動（新ソフトウェアをデプロイ）
         ② 全インスタンス正常確認
         ③ ALB のターゲットグループをグリーンに切り替え → 瞬時に完了

❌ 間違い：先に切り替え → 後でデプロイ（未完成環境にトラフィックが流れる）
```

#### ALB 切り替え vs DNS 切り替え
| 方法 | 速度 | 問題点 |
|---|---|---|
| **ALB ターゲットグループ切り替え** ✅ | 瞬時 | なし |
| Route 53 DNS 変更 ❌ | TTL 分の遅延 | キャッシュ問題・ロールバックも遅い |

#### ブルー/グリーンの基本原則
- ブルー環境（現行）には絶対に手を加えない
- 問題発生時は ALB をブルーに戻すだけでロールバック完了

### 罠
| 選択肢 | なぜ間違い |
|---|---|
| 切り替え → デプロイの順 | 未完成環境にトラフィックが流れる |
| ブルー環境に直接変更 | 環境独立性が失われロールバック不可 |
| Route 53 で DNS 変更 | ALB があるので不要。TTL 遅延が発生 |

### 覚え方
> **Blue/Green = 「まず作って、後で切り替え」「ロールバックは切り戻すだけ」**  
> ALB ターゲットグループ切り替え ＞ DNS 変更（速さ・確実性で優位）

## 問題44：CodeDeploy AllowTraffic 失敗のトラブルシューティング ⭐ 難易度：中
**結果：不正解**

### 一言でいうと
「ログに理由がない = CodeDeploy 管外での失敗 → ALB ヘルスチェック設定ミスが原因」

### カテゴリ別ポイント

#### AllowTraffic ステージとは
```
AllowTraffic（最終段階）
  → インスタンスを ALB ターゲットグループに追加
  → ALB のヘルスチェックが Healthy になるまで待つ
  → タイムアウト → デプロイ失敗（ログには何も残らない）
```

#### 「ログに理由がない」が最大のヒント
| 失敗原因 | ログの状態 |
|---|---|
| ALB ヘルスチェック設定ミス ✅ | ログなし（ALB 側で失敗するため） |
| appspec.yml スクリプトエラー ❌ | スクリプト失敗のログが残る |
| IAM 権限不足 ❌ | AccessDenied エラーログが残る |
| SSM Agent 未インストール ❌ | BeforeInstall で失敗。AllowTraffic まで進まない |

### 罠
| 選択肢 | なぜ間違い |
|---|---|
| CodeDeploy エージェント未インストール | AllowTraffic より前の段階で失敗する |
| appspec.yml スクリプトエラー | ログにエラーが記録される |
| IAM 権限不足 | AccessDenied が明確にログに残る |

### 覚え方
> **「ログに何もない」→ 外部サービス（ALB）側が原因**  
> AllowTraffic = ALB がインスタンスを Healthy と認めるまで待つステージ
