## 問題25：Lambda カナリアデプロイ ＋ SAM ＋ CI/CD ⭐⭐ 難易度：高
**結果：不正解**

### 一言でいうと
「Lambda カナリアデプロイは SAM に任せる。CI/CD は CodePipeline + CodeBuild で組む」

### カテゴリ別ポイント

#### SAM（デプロイ戦略）
- `AWS::Serverless::Function` で Lambda を定義
- `AutoPublishAlias` ＋ `DeploymentPreference` の2プロパティを設定するだけでカナリア完成
- SAM が内部で Lambda Alias / CodeDeploy を自動管理する → 手動設定不要

#### CI/CD パイプライン
| 役割 | サービス |
|---|---|
| ソース管理 | CodeCommit |
| パイプライン制御 | CodePipeline |
| ビルド・デプロイ実行 | CodeBuild ＋ buildspec.yml |

#### 監視・ロールバック
- 各 Lambda 関数に個別の CloudWatch アラームを作成（`AWS/Lambda` の `Errors` メトリクス）
- SAM の DeploymentPreference がアラームを検知して自動ロールバック実行

### 罠
| 選択肢 | なぜ間違い |
|---|---|
| CodeDeploy 手動作成 ＋ appspec.yml | SAM が自動生成・管理するため不要（二重管理） |
| CloudWatch 複合アラーム | 粒度が粗く関数単位の精密監視ができない |

### 全体構造
```
CodeCommit → CodePipeline → CodeBuild（sam deploy）
                                  ↓
                             SAM（Lambda更新）
                                  ↓
                         カナリア（10%テスト）
                                  ↓
                    CloudWatch アラーム（エラー監視）
                                  ↓
                            自動ロールバック
```

### 覚え方
> SAM = 「カナリア＋ロールバック」を1行で書ける魔法  
> CodeDeploy・appspec.yml の手動設定は SAM では不要

## 問題31：ブルー/グリーンデプロイ（ALB ターゲットグループ切り替え） ⭐ 難易度：低
**結果：正解**

### 一言でいうと
「ブルー/グリーンの切り替えは ALB ターゲットグループで。手順は必ず"デプロイ→切り替え"の順」

### カテゴリ別ポイント

#### 正しい手順（順序が命）
```
✅ 正解：① グリーン環境でローリング再起動（新ソフトウェアをデプロイ）
         ② 全インスタンス正常確認
         ③ ALB のターゲットグループをグリーンに切り替え → 瞬時に完了

❌ 間違い：先に切り替え → 後でデプロイ（未完成環境にトラフィックが流れる）
```

#### ALB 切り替え vs DNS 切り替え
| 方法 | 速度 | 問題点 |
|---|---|---|
| **ALB ターゲットグループ切り替え** ✅ | 瞬時 | なし |
| Route 53 DNS 変更 ❌ | TTL 分の遅延 | キャッシュ問題・ロールバックも遅い |

#### ブルー/グリーンの基本原則
- ブルー環境（現行）には絶対に手を加えない
- 問題発生時は ALB をブルーに戻すだけでロールバック完了

### 罠
| 選択肢 | なぜ間違い |
|---|---|
| 切り替え → デプロイの順 | 未完成環境にトラフィックが流れる |
| ブルー環境に直接変更 | 環境独立性が失われロールバック不可 |
| Route 53 で DNS 変更 | ALB があるので不要。TTL 遅延が発生 |

### 覚え方
> **Blue/Green = 「まず作って、後で切り替え」「ロールバックは切り戻すだけ」**  
> ALB ターゲットグループ切り替え ＞ DNS 変更（速さ・確実性で優位）

## 問題44：CodeDeploy AllowTraffic 失敗のトラブルシューティング ⭐ 難易度：中
**結果：不正解**

### 一言でいうと
「ログに理由がない = CodeDeploy 管外での失敗 → ALB ヘルスチェック設定ミスが原因」

### カテゴリ別ポイント

#### AllowTraffic ステージとは
```
AllowTraffic（最終段階）
  → インスタンスを ALB ターゲットグループに追加
  → ALB のヘルスチェックが Healthy になるまで待つ
  → タイムアウト → デプロイ失敗（ログには何も残らない）
```

#### 「ログに理由がない」が最大のヒント
| 失敗原因 | ログの状態 |
|---|---|
| ALB ヘルスチェック設定ミス ✅ | ログなし（ALB 側で失敗するため） |
| appspec.yml スクリプトエラー ❌ | スクリプト失敗のログが残る |
| IAM 権限不足 ❌ | AccessDenied エラーログが残る |
| SSM Agent 未インストール ❌ | BeforeInstall で失敗。AllowTraffic まで進まない |

### 罠
| 選択肢 | なぜ間違い |
|---|---|
| CodeDeploy エージェント未インストール | AllowTraffic より前の段階で失敗する |
| appspec.yml スクリプトエラー | ログにエラーが記録される |
| IAM 権限不足 | AccessDenied が明確にログに残る |

### 覚え方
> **「ログに何もない」→ 外部サービス（ALB）側が原因**  
> AllowTraffic = ALB がインスタンスを Healthy と認めるまで待つステージ

## 問題53：CodeDeploy Lambda フックの実行順序（BeforeAllowTraffic） ⭐ 難易度：中
**結果：正解**

### 一言でいうと
「トラフィック切り替え前にDB確認したい → BeforeAllowTraffic 一択」

### カテゴリ別ポイント

#### Lambda × CodeDeploy のフック実行順序
```
① BeforeAllowTraffic  ← DB確認はここ（トラフィック切り替え前）
② トラフィック切り替え
③ AfterAllowTraffic
④ ValidateService     ← デプロイ完了後の動作確認
```

#### 各フックの役割
| フック | タイミング | 用途 |
|---|---|---|
| **BeforeAllowTraffic** ✅ | 切り替え前 | 前提条件チェック（DB更新完了確認等） |
| AfterAllowTraffic ❌ | 切り替え後 | 手遅れ。既にトラフィックが流れている |
| ValidateService ❌ | 完了後 | 事後確認。エラーは既に発生した後 |
| BeforeInstall ❌ | デプロイ前 | 切り替えまで時間があるので不確実 |

### 覚え方
> **「Before = 前、After = 後」**  
> フック失敗 → 自動ロールバックが走る（安全網）

## 問題58：CodePipeline の並列実行（runOrder） ⭐ 難易度：低
**結果：正解**

### 一言でいうと
「パイプラインを速くしたい → runOrder を同じ値にして並列実行」

### カテゴリ別ポイント

#### runOrder の仕組み
```
同じ runOrder 値 → 並列実行（同時に実行）
違う runOrder 値 → 順次実行（前が終わってから次へ）
```

#### 効果
```
順次実行（5関数）→ 1関数ずつ処理 → 5倍の時間
並列実行（runOrder同値）→ 同時処理 → 理論上1/5の時間
```

#### 他の選択肢が不適切な理由
| 選択肢 | なぜダメ |
|---|---|
| VPC内でCodeBuild | ENI作成でむしろ遅くなる。セキュリティ目的の設定 |
| 計算タイプを上げる | 個別は速くなるが順次実行の構造は変わらない |

### 覚え方
> **「パイプライン並列化 = runOrder を同じ値」**  
> VPC設定はパフォーマンス向上ではなくセキュリティ目的

## 問題61：クロスアカウント CodePipeline の KMS ＋ S3 アクセス拒否 ⭐⭐ 難易度：高
**結果：不正解**

### 一言でいうと
「クロスアカウントアクセス = IAMロール ＋ S3バケットポリシー ＋ KMSキーポリシーの全セット」

### カテゴリ別ポイント

#### 2層の許可が必要（ここで選択ミス）
```
✅ 開発・本番アカウントに IAMロール作成（S3読み取り ＋ KMS復号権限）
✅ ビルドアカウントの S3バケットポリシーで他アカウントのロールを明示的に許可
✅ カスタマーマネージドKMSキーでキーポリシーを設定（他アカウントに復号権限）

❌ IAMロールだけ → S3バケットポリシーがないと拒否される
```

#### AWSマネージドキー vs カスタマーマネージドキー
| | AWSマネージド | カスタマーマネージド |
|---|---|---|
| キーポリシー変更 | ❌ 不可 | ✅ 自由に設定 |
| クロスアカウント | ❌ 対応不可 | ✅ 対応可 |

### 覚え方
> **「IAMロールに権限があるのにアクセス拒否 → リソースポリシー（S3バケットポリシー等）が欠けている」**  
> クロスアカウントは「呼ぶ側（IAM）」と「呼ばれる側（リソースポリシー）」の両方が必要

## 問題75：CodeBuild から ECR への認証（aws ecr get-login-password） ⭐ 難易度：中
**結果：不正解**

### 一言でいうと
「ECRにpushする前に `aws ecr get-login-password` → `docker login` が必須」

### カテゴリ別ポイント

#### S3 vs ECR のアクセスの違い
```
S3 → IAMロール権限があればそのままアクセスできる（Docker認証不要）
ECR → IAMロール権限 ＋ Docker認証（get-login-password）が両方必要
```

#### buildspec.yml の正しい構成
```yaml
pre_build:
  commands:
    - aws ecr get-login-password --region $REGION \
        | docker login --username AWS --password-stdin $ECR_URI
build:
  commands:
    - docker build / tag / push
```

#### なぜ Secrets Manager が間違いか
- ECRトークンは12時間有効の一時トークン（動的）
- Secrets Manager は静的な認証情報の管理用
- → 毎回 get-login-password で取得する必要がある

### 覚え方
> **「IAMロールに ecr:* 権限があっても Docker 認証は別途必要」**  
> ECR = IAM権限 ＋ `get-login-password` → `docker login` の2段階

## 問題76：クロスアカウント EKS アクセス（IAM 信頼関係 ＋ aws-auth ConfigMap） ⭐⭐ 難易度：高
**結果：不正解**

### 一言でいうと
「EKSクロスアカウントアクセス = IAM信頼関係（アプリ側がDevOpsを信頼）＋ aws-auth ConfigMap でKubernetes権限マッピング」

### カテゴリ別ポイント

#### EKS の2段階認証（ここが特有）
```
① IAM 認証：DevOpsアカウントのCodeBuildがアプリのIAMロールをAssumeRole
② Kubernetes RBAC：aws-auth ConfigMap でIAMロール→K8s権限にマッピング
→ どちらが欠けても Unauthorized になる
```

#### 信頼関係の向き（Q54と同じパターン）
```
✅ アプリケーションアカウントのロールが DevOps アカウントを信頼
❌ Control Tower 管理アカウントを経由する必要はない
```

### 覚え方
> **「EKS = IAM信頼関係 ＋ aws-auth ConfigMap の2段階」**  
> Control Tower は組織管理用。クロスアカウントの直接アクセスには不要

## 問題13（P2）：ECS ALB ヘルスチェック失敗ループの診断 ⭐ 難易度：低
**結果：不正解**

### 一言でいうと
「新イメージに更新後にタスクが再起動ループ = ヘルスチェック猶予期間（Grace Period）を延長」

### カテゴリ別ポイント

#### 猶予期間とは
```
タスク起動後に「最初のヘルスチェックを待つ時間」
新イメージで初期化が遅くなった場合は延長が必要
```

### 罠
| 選択肢 | なぜダメ |
|---|---|
| 最低正常率を引き上げる | デプロイ中の最小タスク数制御。根本原因を解決しない |
| ヘルスチェック間隔を短くする | より頻繁にチェック → 逆効果 |

### 覚え方
> **「新イメージで再起動ループ → 猶予期間（Grace Period）を延長」**  
> 猶予期間 = タスク起動後にヘルスチェックを「待ってあげる時間」

## 問題17（P2）：CodeArtifact 組織全体パッケージ管理 ⭐ 難易度：中
**結果：不正解**

### 一言でいうと
「共有ドメイン ＋ アップストリームリポジトリ ＋ 各チームにリポジトリ作成権限」の3セット

### カテゴリ別ポイント

#### 正解の3つ
```
① 共有サービスアカウントにドメインを作成 ＋ 組織全体にCreateRepository権限
② 共有リポジトリをアップストリームとして設定（チームのリポジトリから参照）
③ 各チームのアカウントにリポジトリを作成し、読み書き権限を付与
```

#### アップストリームリポジトリの仕組み
```
チームのリポジトリにパッケージなし → 自動でアップストリームから取得
→ 個別コピー不要・バージョン管理が一元化
```

### 罠（選んだ答え）
- チーム間リソースポリシー → N チームで N×(N-1) 個の設定が必要。スケールしない

### 覚え方
> **「共有ライブラリ = アップストリームリポジトリ」**  
> チーム間の直接依存ではなく、中央の共有リポジトリ経由で取得
