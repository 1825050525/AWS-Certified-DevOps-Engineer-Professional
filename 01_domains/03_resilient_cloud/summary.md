## 問題26：Auto Scaling ウォームプール ＋ ライフサイクルフック ⭐ 難易度：中
**結果：正解**

### 一言でいうと
「起動時間短縮 ＋ コスト効率の両立 → ウォームプールの Stopped 状態一択」

### カテゴリ別ポイント

#### ウォームプールの状態比較（ここが核心）
| 状態 | 料金 | 起動速度 | 用途 |
|---|---|---|---|
| **Stopped** ✅ | EBS 料金のみ | 速い（データ取得済み） | コスト効率重視 |
| Running | フル料金（CPU/メモリ等） | 速い | 超厳格な応答要件 |
| Standby | フル料金 | 変わらず | メンテナンス用 |

#### ライフサイクルフック
- `autoscaling:EC2_INSTANCE_LAUNCHING` を設定
- ウォームプールから復帰後、差分データ取得・最終チェックが完了してから InService に移行
- `CompleteLifecycleAction` でフック完了を通知 → トラフィック受付開始

#### 問題の根本原因と解決策
```
【問題】S3データが増加 → 起動時のダウンロード時間が増加

【解決】ウォームプール（Stopped）
  → インスタンスが既にデータ取得済みで待機
  → スケールアウト時は差分だけ更新してすぐInService
```

### 罠
| 選択肢 | 問題点 |
|---|---|
| ウォームプール Running 状態 | 起動速度は同じだがコスト大（フル課金） |
| 最大インスタンス数を増やす | 常時 Running で高コスト＋S3 DL 問題も未解決 |

### 覚え方
> ウォームプール = **Stopped が正解**（Running は過剰・Standby はメンテ用）  
> ライフサイクルフック = 「準備完了を自分で申告してからトラフィックを受ける」仕組み

## 問題28：マルチリージョン遅延削減（Route 53 ＋ ALB ＋ DynamoDB） ⭐ 難易度：中
**結果：不正解**

### 一言でいうと
「遅延削減 = リージョン分散 ＋ レイテンシーベースルーティング ＋ グローバルテーブル」

### カテゴリ別ポイント

#### Route 53 ルーティング（ここで選択ミス）
| ポリシー | 用途 | 今回の要件 |
|---|---|---|
| **レイテンシーベース** ✅ | 実測遅延が最小のリージョンへ誘導 | 遅延削減に最適 |
| フェイルオーバー ❌ | 障害時にプライマリ→セカンダリ切替 | 通常時は片方しか使わない |

#### ALB / Auto Scaling のリージョン構成
```
❌ 誤解：「グローバルALB」「グローバルAuto Scaling」は存在しない

✅ 正解：新リージョンに ALB + Auto Scaling グループを個別に作成する
         → Route 53 がリージョン間のトラフィックを振り分ける
```

#### DynamoDB のグローバル対応
| 手法 | 評価 |
|---|---|
| 手動クロスリージョンレプリケーション ❌ | 同期複雑・競合解決困難・運用負荷大 |
| **グローバルテーブル** ✅ | マルチマスター双方向レプリケーション・自動競合解決 |

### 罠
| 選択肢 | なぜ間違い |
|---|---|
| フェイルオーバールーティング | 通常時は片方リージョンのみ使用。遅延削減効果なし |
| グローバルALB/ASGをセットアップ | そのようなサービスは存在しない |

### 覚え方
> 遅延削減の3点セット：**新リージョンにALB+ASG** ＋ **Route 53レイテンシーベース** ＋ **DynamoDBグローバルテーブル**  
> フェイルオーバー ≠ 遅延削減（これは障害対応用）

## 問題48：S3 クロスリージョン・クロスアカウント レプリケーション（CRR） ⭐ 難易度：中
**結果：正解**

### 一言でいうと
「CRRの設定はソース側が能動・ターゲット側は受動。IAMロールとルールはソース、バケットポリシーはターゲット」

### カテゴリ別ポイント

#### 設定の役割分担（ここが核心）
```
ソースアカウント（能動側）
  ✅ レプリケーション IAM ロール作成
  ✅ ソースバケットにレプリケーションルール定義
  ❌ ソースバケットポリシーの修正は不要（同一アカウント内なので）

ターゲットアカウント（受動側）
  ✅ ターゲットバケットポリシーを修正（ソースのIAMロールに書き込み許可）
  ❌ ターゲット側にIAMロールは不要
  ❌ ターゲット側でルール定義は不要
```

#### 既存オブジェクトの注意点
- CRRルール設定後も既存オブジェクトは対象外
- 既存オブジェクトを移したい場合 → **S3 Batch Replication**

### 罠
| 選択肢 | なぜ間違い |
|---|---|
| ソースバケットポリシーを修正 | 同一アカウント内のIAMロールなので不要 |
| ターゲット側にIAMロールを作成 | 処理主体はソース側のみ |
| ターゲット側でルール定義 | ルールはソースバケット側だけで定義する |

### 覚え方
> **「ルール＆IAMロールはソース、バケットポリシーはターゲット」**  
> クロスアカウントの書き込み許可 = ターゲットバケットポリシーの役割

## 問題60：複数VPC間プライベート通信（AWS PrivateLink） ⭐⭐ 難易度：高
**結果：不正解**

### 一言でいうと
「20のVPC ＋ 同一CIDR ＋ 変更最小 = PrivateLink ＋ NLB」

### カテゴリ別ポイント

#### なぜ VPC ピアリングがダメか（選んだ答え）
```
① CIDR が全て同じ（192.168.0.0/22）→ IPアドレス重複 → ピアリング不可
② 20 VPC → 190本のピアリングが必要 → ルートテーブル管理が地獄
③ 新サービス追加のたびに全VPCのルートテーブル更新が必要
```

#### PrivateLink が最適な理由
```
✅ CIDR 重複でも使える（ENI 経由のため）
✅ 既存環境への変更ほぼなし（NLB追加 ＋ エンドポイント作成のみ）
✅ トラフィックはAWSバックボーン内に閉じる（インターネット不使用）
✅ ルートテーブル変更不要
```

#### 構成
```
提供側：既存VPC ＋ NLB（エンドポイントサービスとして登録）
利用側：VPCエンドポイントを作成してサブスクライブ
```

### 罠
| 選択肢 | なぜダメ |
|---|---|
| VPCピアリング | 同一CIDRでは不可。190本の接続でスケールしない |
| 共有サブネット（RAM） | 既存VPCから丸ごと移行が必要 |
| Transit Gateway | ルートテーブル変更が必要。同一CIDR問題が残る |

### 覚え方
> **「同一CIDR ＋ 多数VPC ＋ 変更最小 → PrivateLink 一択」**  
> PrivateLink = 「提供側にNLB、利用側にVPCエンドポイント」

## 問題12（P2）：S3 レプリケーション失敗の大規模再試行（S3 Batch Operations） ⭐ 難易度：中
**結果：不正解**

### 一言でいうと
「大規模S3レプリケーション再試行 = S3 Batch Operations（Lambda でダウンロードしない）」

### カテゴリ別ポイント

#### Lambda で PutObject がダメな理由
```
Lambda の制限：実行時間15分 / /tmp 10GB / 同時実行制限
→ 大規模データのダウンロード→アップロードはこれらに引っかかる
```

#### S3 Batch Operations が最適
```
✅ 数十億オブジェクトでもスケール
✅ S3ReplicateObject = S3 ネイティブのレプリケーションを再実行（ダウンロード不要）
✅ Lambda はジョブ作成のトリガー役のみ
```

### 覚え方
> **「大規模S3の一括処理 → S3 Batch Operations」**  
> Lambda で直接処理するのは小規模向け。「大規模データ」とあれば Batch Operations を選ぶ
